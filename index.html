DFS review

  • Clients
    • send requests to file service
  • Caching
    • improve performance (seen by clients) and scalability (supported by servers)
  • Servers
    • own and manage state (files)
    • provide service (file access)

Peer Distributed Applications

  • Commonly used in big data services/systems
  • Each node
    • “own” state
    • provides service
    • all nodes are “peers” - has local accesses, provide accesses to other nodes and could access other nodes
  • Some nodes could still be managers that are more centralized

Distributed Shared Memory (DSM)

  • Service manages memories across all machines
  • Each node
    • “owns” state => memory
    • provides service
      a. memory reads/writes from any node
      b. consistency protocols
  • Permits scaling beyond single machine memory limits
  • Benefits: more “shared” memory at lower cost
  • Downside: slower overall memory access

Hardware v.s. Software DSM

Hardware Supported - EXPENSIVE

  • relies on interconnect
  • OS manages larger physical memory
  • NICs translate remote Main Memory accesses to messages
  • NICs involved in all aspects of Main Memory management, support atomics …

Software Supported

  • Everything done by software
  • Can be done by OS, or language runtime

DSM design

Sharing Granularity

  • Cache line granularity? - updating on every cache line change?
    • overheads too high for DSM, and slow the performance a lot
  • Variable granularity? - good for developers, but still high overheads as we have lots of variables
  • Page granularity (OS-level)
  • Object granularity (Language runtime)
  • When using the larger granularity such as Page and Object, be aware of false sharing: e.g. x and y are on same page. Programmer should be aware of the layout of variables/objects, or rely on some smart compilers

Access Algorithm

  • Application access algorithm
    • single reader/single writer (SRSW) - DSM simply needs to provide access to remote/shared memories, no consistensy/sharing related challenges
    • Multiple readers/single writer (MRSW)
    • Multiple readers/multiple writers (MRMW)
    • The latter two need to think about how to retrieve latest data for each read on all nodes, and how to ensure consistency among writes across all nodes

Performance consideration

  • Metrics: access latency
  • Achieving low latency through
    a. Migration: 
    - makes sense for SRSW
- requires data movement
- overhead if only single read/write
    b. Replication
    - states are copied to multiple nodes
- more general
- requires consistency management

Consistency management

  • Push invalidations when data is written to - proactive, eager, pessimistic
  • Pull modification info periodically - reactive, lazy, optimistic
  • When these methods get triggered depends on the consistency model for the shared state

DSM Architecture

Page-based DSM architecture, OS supported

  • Distributed nodes, each with own local main memory contribution
  • Pool of pages from all nodes
  • Each page has ID, page frame number
  • If MRMW
    • need local caches for performance
    • home (or manager) node drives coherence operations
    • all nodes are responsible for part of distibuted memory management
    • Acting as both server and client
  • home node
    • Keeps track of states: pages accessed, modifications, caching enabled/disabled, locked, etc
    • current “owner”, currently owns the page, that could control all state updates and drive consistency operations (owner may not equal home node)
  • Explicit replicas
    • for load balancing, performance or reliability
    • controlled by home node

Indexing Distributed State (memory)

  • Maintain some DSM metadata
  • Each page (object) has
    • address == node Id + page frame number
    • node Id == “home” node
  • Global map (replicated across all nodes)
    • object (page) id => manager node id
    • manager map available on each node
    • could be implemented through global mapping table: object id => index into mapping table => manager node
  • Metadata for local pages (partitioned)
    • Per-page meadata is distributed across managers

Implementing DSMs

  • DSM must “intercept” accesses to DSM state
    • to send remote messages requesting access
    • to trigger coherence messages
  • Overheads should be avoided for local, non-shared state (pages)
    => dynamically “engage” and “disengage” DSM when necessary

solution

  • Use hardware MMU support
    • trap in OS if mapping invalid or access not permitted
    • remote address mapping => trap and pass to DSM to send message
    • cached content => trap and pass to DSM to perform necessary coherence operations
    • Other MMU information usefule (e.g. dirty page)

Consistency model

What is Consistency model

Agreement between memory(state) and upper software layers: memory behaves correctly if and only if software follows specific rules

Memory (state) guaranteees to behave correctly…

- access ordering
- propagation/visibility of updates

This is general and could be apply to multiple threads on one machine

Notations:

Strict Consistency

  • updates visible everywhere immediately
  • guarantees that every node will see all writes happened in the system in the exact same order
  • In practice:
    • even on single SMP no guarantees on order without extra locking and synchronization
    • in distributed systems, latency & message reorder/loss make this even harder -> impossible to guarantee

Sequential Consistency

  • As long as the order matches some possible ordering of write operations, its ok
  • Memory updates from different processors may be arbitrarily interleaved
  • But all processes should see the same ordering (interleaving)
  • Constraint: operations from same process always appear in order they were issued

Causal Consistency

  • Different orders will be NOT OK when there are depenencies between orders of operations
  • Detect causally reltated writes and order them in same orders

Weak Consistency

  • Provides synchronization points - operations that are available:
    • all updates prior to a sync point will be visible
    • no guarantee what happens in between
    • has to be performed by both the process who performs the update and the process who wants to see the update
  • Variation
    • single sync operation (sync)
    • separate sync per subset of state (page)
    • separate entry/acquire (polling) v.s. exit/release(pushing) operations
  • Benefits: limit data movement & coherence operations
  • Downside: maintain extra state for additional operations

Preview

Visual Metaphor

Features/Functionalities Distributed File System Storage Facility in a shop
Accessed via well-defined interface Access via Virtual File System (VFS) Boss looks at stoage reports
Focus on consistent state Tracking state, file updates, cache coherence … Correctly determine inventory
Mixed distribution models possible Replicated v.s. partitioned, peer-like systems Storage only, storage and processing, toy-specific

DFS Models

  • Client/Server on different machines
  • File server distributed on multiple machines
    • replicated (each server: all files)
    • partitioned (each server: part of files) - more scalable
    • both (files partitioned; each partition replicated) - a lot of big data file systems use this approach
  • Files stored on the served from all machines (peers) - blurred distintion between clients and servers

Replication

  • Each machine holds all files (exact replica)
  • Benefits
    • load balancing: request across all machines -> better performance
    • availability: systems are more available
    • fault tolerance
  • Downsides
    • Writes become more complex. solution could be:
      a. synchronously write to all replicas (this will result in long write)
      b. write to one, then propagated to others; replicas then need to be reconsiled

Partitioning

  • Each machine has subset of files
  • Benefits:
    • Availability v.s. single server DFS
    • Scalability with file system size
    • Single file writes simpler

  • Downsides

    • On failure, lose portion of data
    • Load balancing harder

  • Could combine both - replicate each partition

Remote File service

Extreme 1: Upload/Download

  • like FTP, SVN
  • Benefits: local reads/writes at client
  • Downside:
    • entire file download/upload even for small accesses
    • server gives up control

Extreme 2: True Remote File access

  • every access to remote file, nothing done locally
  • Benefits: file accesses centralized, easy to reason about consistency
  • Downside:
    • every file operation pays network cost
    • limits server scalability because every file operation is against server

More Practical Remote File Access with Caching (Compromise)

  • Allow clients to store parts of files locally (as blocks), or prefetch part of files, just like locally prefetching files from disk into memory.
    • Benefit:
      a. Low latency on file opeartions
      b. Server load reduced => more scalable
  • Force clients to interact with server frequently, to maintain consistency:
    • Clients need to notify changes they’ve made to files
    • Clients also need to sync with server to see if any files in caches have been modified by others
    • Benefits:
      a. Server has insights into what clients are doing
      b. Server has control into which accesses can be permitted => easier to maintain consistency
  • Downside: Server more complex, requires different file sharing semantics.

Stateless v.s. Stateful File Server

Stateless == keeps no state

  • No information about clients/files at all, each request should describe itself
  • Ok with extreme models, but cannot support practical models
  • Downside
    • Cannot support caching and consistency management
    • Every request is self-contained/self-described => more bits transferred
  • Benefits
    • No resources are used on server side (CPU/MM)
    • On Failure, just restart, as there is no state

Stateful == keeps client state

  • Needed for Pratical Models
  • Will track information for files/clients and permissions, etc.
  • Benefits
    • Can support locking, caching, incremental operations like reading the next X bytes of a file
  • Downsides
    • On failure, need checkpointing and recovery mechanisms
    • Overheads to maintain state and consistency -> depends on caching mechanism and consistency protocol

Caching State in a DFS (Optimization)

  • Locally clients maintain portion of states (e.g. file blocks)
  • Locally clients perform operations on cached state (e.g. open/read/write)
  • Requires coherence mechanisms - details depend on file sharing semantics
  • Where could files or file blocks be cached?
    • In client memory
    • On client storage devices (HDD/SSD…)
    • In buffer cache in memory on server

File Sharing Semantics on a DFS

Comparison between single machine file sharing and DFS file sharing

Single machine (UNIX Semantic)

  • Changes could be seen by all other processes, because they are accessing the same memory bucket, even when changes are not pushed to disk yet

DFS

  • Changes cannot be seen by other clients even when the changes are pushed to server immediately, because there are still latency/delay
  • We cannot determine how much we should delay on each client in order to get the latest valid file
  • Will usually sacrifice some consistency

Solutions

Session semantics

  • Write-back to server on close(), update on open() (i.e. doesnt use the cache version)
  • A session is between open-close
  • Achieves consistency, but clients might be out of sync with others concurrently.
  • Easy to reason, but may be insufficient

Periodic Updates

  • Client writes-back periodically -> clients have a “lease” on cached data (not exclusive necessarily)
  • Server invalidates periodically -> provides bounds on “inconsistency”
  • Augment with **flush()/sync() API

Immutable files

  • never modify, new files created

Transactions

  • File system should export some APIs to make all changes atomic

Access Patterns: Files v.s. Directories

  • Choose different policies for each
    • eg1: session-semantics for files, UNIX for directories
    • eg2: less frequent write-back for files than directories

Network File system

NFS Versions

  • NFSv3 - stateless, NFSv4 stateful
  • Caching
    • Session-based (non-concurrent), on close, flush to server, and on open, check server
    • Periodic Updates: default - 3 sec for files, 30 secs for directories
    • NSFv4 => delegation to client for a period of tiem (avoid “update checks”)
  • Locking
    • lease-based, lease the lock to client for a number of time

Sprite Distributed File System

  • Reference: Caching in the Sprite Network File System

File Access Parttern (Workload) analysis

  • 33% of file accesses are writes => caching is ok, as we have 66% of read access, but write-through is not sufficient
  • 75% of files are open less than 0.5 sec
  • 90% of files are open less than 10 sec => session semantics still too high overheaded
  • 20 - 30% of new data deleted within 30sec
  • 50% of new data deleted within 5 minutes => write-back on close not really necessary
  • File sharing is rare => no need to optimize for concurrent access, but must support it

From Analysis to Design

  • Cache with write-back
    • every 30sec write-back blocks that have NOT been modified for the last 30sec, as a recently modified file tends to be continuously modified
    • When other client opens file => get dirty blocks from others
  • Open goes to server; directories not cached
  • On “concurrent write” => disable caching
  • Sharing semantics
    • Sequential write sharing: caching and sequential semantics
    • Concurrent write sharing: no caching

File Access Operations in Sprite

  • R1, …, Rn Readers, w1 writer
    • all open() go through server
    • all clients cache blocks
    • writer keeps timestamps for each modified block, and check consistency with server when open
  • w2 sequential writer shows up (sequential sharing)
    • server contacts last writer for dirty blocks
    • if w1 was closed, update version
    • w2 can now cache file
  • w3 concurrent writer shows up (concurrent sharing)
    • server contacts last writer to gather dirty blocks
    • since w2 hasn’t closed => disable caching for everybody
    • When either w2 or w3 closes the file, server will see that and mark the file as cacheable
  • Info on server/client

Propositional Logic

Given B E A M J standing for some symbol, so that

  • E or B implies A: (E v B) -> A
  • A implies J and M: A -> (J ^ M)
  • J if and only if M: J <–>M

Terminology

  • A statement is valid if it is true in every possible model
  • A statement is satisfiable if it is true in some models

Propositional Logic Limitations

  1. Can only handle true/false value, cannot handle Uncertainty
  2. Cannot talk about Objects which have properties, and cannot talk about relationships between objects
  3. No shortcut to succinctly talk about complex world

Type of Constraint

Unary Constraint

  • Restrict the value of a single variable.
  • Example, v1 != value

Binary Constraint

  • Relates two variables.
  • Example, v1 != v2
  • A binary CSP is the one with only binary constraints, and it can be represented as a constraint graph

Global Constraint

  • Involving an arbitrary number of variables (Note: Not all varaibles should be involved)
  • Example: Alldiff, meaning variables involved should have different values

Constraint Hypergraph

  • Consists of
    1. Ordinary nodes: usually circles, representing variable
    2. Hypernodes: usually square, representing n-ary constraints

Backtracking Search

Improving Backtracking efficiency

Least Constraining Value

choose the variable that rules out the fewest values in the remaining variables

Minimum Remaining Values

choose the variable with the fewest legal values; when encoutering tie, choose the one with most constraints (this will lead to problems, if any, sooner rather than delaying them)

Forward Checking

For each of the unassigned variable, we will keep track of the remaining legal values; if we ever encountered a scenario such that a variable does not have any legal values remained, we stop and backup

It is an early waning system basically

Constraint Propagation and Arc Consistency

Node Consistency

A single variable is called node consistent if all the values in the variable’s domain satisfy the variable’s unary constraints

A network is node consistent if every variable in the network is node-consistent

Arc Consistency

A variable is arc consistent if every value in its domain satisfies the variable’s binary constraints.

More formally, Xi is arc consistent with respect to another variable Xj if for every value in the current domain Di there is some values in the domain Dj that satisfies the binary constraint on the arc (Xi, Xj)

A network is arc consistent if every variable is arc consistent with every other variable

  • Most popular algorithm to solve arc consistency

NanoDegree AI - Simulated Annealing

Posted on 2017-04-14

Introduction

The philosophy behind this is to start from simple, and add a little intelligence only when necessary, like iteratively climbing up a hill. However, there could be local maximum/minimum that we need to avoid

Typical Example

n-Queen problems: moving a queen to reduce conflicts(attacks) is a simulation of hill climbing, and when there is no move to reduce conflicts yet the puzzle is not solve, that is a local maximum

Hill Climbing and Local Maximum

Solution

Do a random restart. It is not very efficient, as we need to do a lot of random restart to collect examples and end up with a good result

Improvements

  1. A Taboo Search would be more efficient, which will keep track of spaces you have been (restarted), and only restart at unknown positions
  2. A Variant of Stage Algorithm would also perform better, which will keep a list of all local maximum, and get a general shape of the problem

Other Problems

  • If step size is too small, it might be stuck at the shoulder, and cannot see the upcoming increase. Need to determine when (i.e. how many tries on 0 improvements) to stop
  • If step size is too large, it could possibly skip the maximum, and could sometimes get to infinit loop situation

Simulated Annealing

Annealing

Heating to hight temperature to increase the randomness, and then gradually cool it down to converge to a solution

Simulated Annealing

  • Idea:

  • This will also solve the platue scenario, as deltaE is 0 in that case, it will always move forward

Local Beam search

  • Instead of starting at only one position while performing Simulated Annealing, start at k positions, and keep the k best positions
  • If any of these k best positions reach the goal, we terminate

Genetic Algorithm

An analogy of the natural selection in biology, and uses breeding and mutation to find the optimal solution to a problem

General procedure

  1. Evaluate all randomly selected states (chosen as parents), assign a score to each of them
  2. Based on the score, normalize and calculate the percentage such that the related state will be selected to “breed” children
  3. Select another set of parents based on the percentages calculated in step 2)
  4. Do crossover and mutation to breed children using pairs of the parents
  5. Using the new generations as parents and repeat step 1, till we have a good generation that could optimally fix the problem

Cross-over

Cross over simply means crossing the parents randomly, and use part of them to combine with the other parent, and generate the next generation

Mutation

  • Motivation: critical part or information might be lost or never reached in some cases, so we want to introduce more randomness to bring them back. Just like mutation in biology
  • Each piece of information of the state will have a small but significant chance to be mutated to another piece of information

NanoDegree AI - Intro to Search

Posted on 2017-04-12

Definition of a problem

  • Initial State -> S0
  • Actions (s) -> {a1, a2, a3, …}
  • Result (s, a) -> s’ (a new state)
  • GoalTest(s) -> True|False
  • PathCost (sequence of state, actions and transtions) -> cost value(n)

State

  • Three areas:
    • Frontier
    • Explored
    • Unexplored

General tree search

General Graph Search:

Breadth-First Search

  • Shortest First Search
  • When doing remove.choice(frontier), it will always pick the one nearest to current state first
  • The goal test is passed when we remove a state from frontier
  • Guarantee to find shortest path if there is no/equal path cost

Depth-First Serch

  • Longest First search
  • When doing remove.choice(frontier), it will always pick the one furthest to current state first

Uniform Cost Search:

  • Cheapest First Search
  • When doing remove.choice(frontier), it will always pick the one with lowest total path cost so far first
  • Time wasted in searching every direction in order to locate the goal -> If we have an estimate of the distance from S to G, we could find the goal faster

Greedy Best-First Search:

  • Expand the one has least estimate of distance first
  • It will stick to the correct direction, but will cause problem if there are obstacles. (normally will end up longer path)

A* search

  • Combine with Uniform Cost Search and Greedy Best-First Search: Best estimated total path cost first
  • Always expand the node that has Minimum Value of f = g + h, where
    • g(path) = path cost
    • h(path) = estimated distance from that node to goal
  • Result: Search stratetgy that is the best possible, and finds the shortest path while expanding minimum number of paths possible
  • It is ot guaranteed to find the shortest cost path, as it depends on the heuristic function

Properties of Searches

  • Optimal: is it guaranteed to find the shortest path
    • Breath First Search
    • Uniform Cost Search
  • Completeness: is it guaranteed to find the goal, if there is a goal somewhere (consider infinite tree/graph)
    • Breath First Search
    • Uniform Cost Search

Optimistic Heuristic

  • We say that a heuristic function is optimistic or admissible if it never overestimates distance to goal: h(s) < true cost
  • How to come up with good heuristic function? - Gracefully relax the problem

Problems with search

  • Problem must be
    1. Fully Observable: must be able to see what initial we start out with.
    2. Known: must know the set of available actions to us
    3. Discrete: must be a finite number of actions to choose from
    4. Deterministic: must know the result of taking an action
    5. Static: must be nothing else that can change thet world other than our own action

Implementing Remote Procedure Calls

Why RPC

  • Common steps related to remote Inter Process Communication -> Remote Procedure Calls
  • Simplify the development of cross-address space & cross-machine interactions
  • Benefits
    • Higher-level interface for data movement and communication
    • Error handling
    • Hiding complexities of cross-machine interactions

RPC Requirements

  1. Client/Server Interactions
  2. Procedure Call Interface => sync call semantics
  3. Type Checking
    • Error handling
    • Packet bytes interpretation
  4. Cross-machine conversion. Should hide the differences from programmers
    • e.g. big/little endian; negative numbers
    • One way to solve is to agree with a format, such as nnetwork format
  5. Higher-level Protocol
    • Access control, fault tolerance…
    • Different transport protocols

RPC Structure

Steps

  1. register: server registers procedure, args types, locations, etc…
  2. bind: client finds and bind to desired server
    Client
  3. call: client makes RPC call; control passed to stub, client code blocks
  4. marshal: client stub marshals arguments (serialize args into buffer)
  5. send: client sends message to server
    Server
  6. receive: server receives message; pass msg to server-stub; access control
  7. unmarshal: server stub unmarshals args (extracts args & creates data structures)
  8. actual call: server stub calls local procedure implementation
  9. result: server performs operation and computes result of RPC operation

Interface Definition Language

Reasoning

  • Server and Client can be developed separately
  • But do need an agreement to support
    • client side bind decision. (i.e. which server to bind to, and what operations to perform)
    • runtime to automate stub generation
  • IDL is basically a protocol of how these agreements are expressed

Interface Specification

  • Used to describe the interface the server exports:
    • Procedure name, arg & result types -> pretty much like a function definition
    • version #: helps client identify which server has latest implementation, and is useful when performing upgrades
  • RPC can use IDL that is
    • language-agnostic: XDR in SunRPC
    • language-specific: Java in Java RMI
  • Only for specification, not for implementation

Binding

  • Client determines
    • which server to bind to: service name, version number, etc.
    • how to connect to that server: IP Address, network protocol
  • It will use a Registry, which is pretty much a database of available services
    • Search for service name to find service (which) and contact details (now)
    • Could be distributed: any RPC service can register
    • OR could be machine-specific: only for services running on same machine, and requires client to know machine address
  • Needs naming protocol

Marshalling - add program as an example

  • Provided by RPC System Compiler
  • Encode data and argument in a readable and understandable way

Un-marshalling

  • Provided by RPC System Compiler
  • Interpret the address space and decode data

Features/Functionalities Applications using binding & registries Toy shops use directories of outsourcing services
Who can provide a service? look up registry for image processing shops to outsource assembly
What services do they provide compress, filter, version # -> IDL that will create train carts

What about pointers?

Pointers are pointing to loction in caller address space

Solutions

  1. No pointers!
  2. Serialize pointers; copy referenced (“pointed to”) data structure to send buffer

Summary

  • Binding: How to find the server. (Distributed or machine specific)
  • IDL: How to talk to the server; how to package data. (Language agnostic or language specific)
  • Pointers as arguments: disallow or serialize pointed data
  • Partial failures: Special error notifications

Sun RPC

Design Choices

  • Binding: Assumes server machine is know upfront -> per-machine registry daemon
  • IDL: Language agnostic: XDR (for interface specification and for encoding)
  • Pointers: allowed and serialized
  • Failure: retries; return as much information as possible

Overview

SunRPC Overview

  • client-server via procedure calls
  • Interface specified via XDR (.x file)
  • rpcgen compiler: converts .x to language-specific stubs
  • Server registers with local registry daemon
  • Registry (per-machine): name of service, version, protocol(s), port number…
  • Binding creates handle
    • client uses handle in calls
    • RPC runtime uses handle to track per-client RPC state
  • Coule be on the same or different machines

XDR Example

  • XDR (.x file) describes:
    • datatypes
    • procedures (name, version)
    • service ID
  • Client uses: service name, proc name and version number
  • RPC runtime refers to service id and proc id and version number

Compiling XDR

  • rpcgen compiler: rpcgen -c square.x for C
  • Generates:
    1. square.h: data types and function definitions
    2. square_svc.c: server stub and skeleton (main) - no implementations
      • main: registration/housekeeping
      • square_prog_1: internal code, request parsing, arg marshalling
      • square_proc_1_svc: prototype of actual procedure; must be implemented by developers
    3. square_clnt.c: client stub
      • squareproc_1: wrapper of RPC call to square_proc_1_svc
      • Developers can then do y = squareproc_1(&x, …)
    4. square_xdr.c: common marshalling routines

XDR Compilation Summary

  • rpcgen -c square.x: not thread safe
  • rpcgen -c -M square.x: multithreading safe!

XDR Data Types

  • Default types: char, byte, int, float, etc.
  • Additional XDR types
    • const (#define)
    • hyper (64-bit integer)
    • quadruple (128-bit float)
    • opaque (~C byte): uninterpreted binary data
  • Fixed-length array: e.g. int data[80]
  • Variable-length array: e.g. int data<80>
    • translates to data structure with “len” and “val” fields. “len” is the actual length of the array and “val” points to the base memory
    • Except for string

XDR Routines

  • marshalling/unmarshalling: found in square_xdr.c
  • Cleanup
    • xdr_free()
    • User defined _freeresult procedure, e.g. square_prog_q_freeresult
    • Called after result is returneds

  • Need to encode arguments or results into a bytestream depending on data type
  • Need agreement on this

  • XDR == IDL + encoding
  • XDR Encoding Rules
    • all data types are encoded in multiples of 4 bytes
    • big endian is the transmission standard`

Reference

Virtual Machine Monitors: Current Technologies and Future Trends

What is virtualization

Virtualization allows concurrent execution of multiple OSs (and their applications) on the same physical machine

  • Virtual resources == each OS thinks that it “owns” hardware resources
  • Virtual Machine (VM) == OS + applications + virtual resources (guest domain)
  • Virtualization layer == management of physical hardware (Vitual machine monitor, hypervisor)

Defining Virtualization

A Virtual machine is an efficient, isolated duplicate of the real machine

  • Supported by a Virtual Machine Monitor (VMM)
    1. Provides environment essentially identical with the original machine - fidelity
    2. Programs show at worst only minor decrease in speed - performance
    3. VMM is in complete control of system resources － safety & isolation

Benefits

Consolidation

* Run multiple OS on a single machine
* Decrease cost, improve manageability

Migration

* Availability, reliability

Security, debugging, support for legacy OSs

Virtualization Models

Bare-metal or Hypervisor-based (type1)

  • VMM*hypervisor) manages all hardware resources and supports execution of VMs
  • Privileged, service VM to deal with devices (and other configuration and management task), and runs all drivers

Examples

Xen

  • Opern source or Citrix XenServer
  • domain0 - Privileged VM and domUs- Guest VM
  • Drivers in domain0

ESX (VMware)

  • Many open APIs
  • Drivers in VMM
  • Used to have Linux control core, now remote APIs

Hosted (type2)

  • Host OS owns all hardware
  • Special VMM Module provides hardware interfaces to VMs and deals with VM context switching
  • Can leverage all existing features in Host OS

Examples

KVM (Kernel-basedVM)

  • Based on Linux
  • KVM kernel module + QEMU for hardware virtualization
  • Leverages Linux open-source community

Hardware Protection Levels

Commodity hardware has more than 2 protection levels

e.g. x86 has

  • 4 protection levels (rings)
    ring3: apps
    ring1: OS
    ring0: hypervisor
  • and 2 protection modes (root & non-root)
    • non-root: VMs
      ring3: apps
      ring0: OS
    • root: ring0: hypervisor

Processor Virtualization (Trap and Emulate)

Guest instructions…

- Executed **directly by hardware**, VMM not involved
- For **non-privileged operations**: hardware speeds -> efficiency
- For **privileged operations**: __TRAP__ to hypervisor
- Hypervisor determines what needs to be done:
    a. if **illegal op**: terminate VM
    b. if **legal op**: __emulate__ the behavior the guest OS was expecting from the hardware

Problems with Trap-and-Emulate

When privileged operations fail silently, the hyperevisor doesn’t know and hence would try to do anything; and the OS doesn’t know that hypervisor didn’t do anything, and would assume any operation it requested as successful operations. In Pre 2005 x86 machines, there are 17 such operations

Solutions

Binary Translation

  • Main Idea: rewrite the VM binary to never issue those 17 instructions
  • Commercialized by VMWare
  • Goal: full virtualization == guest OS not modified
  • Approach: Dynamic binary translation
    1. Inspect code blocks to be executed
    2. If needed, translate to alternate instruction sequence
      • e.g., to emulate desired behavior, possibly even avoiding trap
    3. Otherwise, run at hardware speeds
      • Cache translations

Paravirtualization

  • Goal: Performance - give up on unmodified guests
  • Approach: paravirtualization == modify guest so that
    • its knows it’s running virtualized
    • it makes explicit calls to the hypervisor (hypercalls), for those 17 instructions
    • hypercall (~system calls)
      1. Package context info
      2. Specify desired hypercall
      3. Trap to VMM
  • e.g., Xen

Memory Virtualization

Full

  • All guests expect contiguous physical memory, starting at 0
  • Need to distinguish between
    a. Virtual Addresses
    b. Physical Addresses
    c. Machine Addresses
    d. Page frame bumbers
  • Still leverages hardware MMU, TLB, …
  • Option 1:
    • Guest page table: VA -> PA
    • Hypervisor: PA -> MA
    • TOO EXPENSIVE
  • Option 2:
    • Guest Page Tables: VA => PA
    • Hypervisor shadow PT: VA => MA
    • Hypervisor maintains consistence

Paravirtualized

  • Guest aware of virtualization
  • No longer strict requirement on contiguous physical memory starting @ 0
  • Explicitly registers page tables with hypervisor

Device Virtualization

  • For CPUs and memory - less diversity, because ISA (Instruction Set Architecture) has standardization of interface, and it’s up to the hardware manufacturer to implement them
  • For devices - high diversity, and lack of standard specification of device interface and behavior

3 Key Models for Device Virtualization

Passthrough Model

  • Approach: VMM-level driver configures device access permissions
  • Benefits:
    • VM provided with exclusive access to the device
    • VM can directly access the device (VMM - bypass)
  • Downside:
    • Device sharing difficult
    • VMM must have exact type of device as what VM expects
    • Because VM is tied to the device, it breaks the decouple rule and makes migration tricky

Hypervisor-Direct model

  • Approach:
    • VMM interrupts all device accesses
    • Emulate device operations:
      a. Translate to generic I/O Operations
      b. Tranverse VMM-resident I/O stack
      c. Invoke VMM-resident driver
  • Benefits
    • VM decoupled from physical device
    • Sharing, migration beceom easier
  • Downside:
    • Latency of device operations
    • Device driver ecosystem complexityin hypervisor

Split-Device Driver Model

  • Approach: Device access control split between
    • front end driver in guest VM (device API)
    • back end driver in service VM (or host): just regular device driver
    • Modified guest drivers => limited to paravirtualized guests
  • Benefits:
    • Eliminate emulation overhead
    • Allow for better management of shared devices, because we have a back-end device driver centralized across all guest VMs

CPU Device Interactions

CPU to Devices

  • Device access is pretty much accessing memory, and then the PCI bus will route the memory access to related devices. This needs part of the physical memory be mapped for I/O -> use some Base Address Register
  • I/O port: dedicated in/out instructions for device accesses

Devices to CPU

Interrupt

Devices generate interrupts to CPU, and gets handled by oppose

+: Can response as soon as possible
-: Interrupt handling could be expensive

Polling

CPU keeps polling devices by reading their status registers to see if there are actions/data available

+: Convenient time for Operating Systems
-: Delay or CPU overhead

Device access

Programmed I/O (PIO)

  • No additional hardware support. CPU directly issues instructions by writing to the command register of a device, and controls data movement by accessing the data register of a device

Direct Memory Acceess (DMA)

  • Relies on DMA controller
  • CPU still issues instructions directly to command registers
  • CPU interacts with data register via DMA Controller
  • This will result in less steps, but the DMA step is complexed
  • Data buffer used by DMA must be in physical memory untill transfer completes => pinned region (not swappable)

Typical Device Accesses

Operating System Bypass

  • User process can directly access device regs/data
  • Operating System involves in configuration, but then out-of-the-way
  • Needs user-level driver (~library) to connect user process and drivers, usually provide by manufacturer
  • Relies on device features, e.x. sufficient registers so that OS can map some to user processes
  • Operating System retains coarse-grain control

Asynchronous I/O Operations

  • Process continues
  • Later, either
    a. Process checks and retrieves result (polling based)
    b. Process is notified that the operation is completed and results are ready (interupt based)

Virtual File system

Abstractions

  • File == elements on which the VFS operates
  • file descriptor == Operating System representation of file; support different actions (e.x. open, read, write)
  • inode persistent representation of file “index”
    1. list of all data blocks
    2. device, permissions, size, …
  • dentry == directory entry, corresponds to single path component, only in memory maintained by OS
  • dentry cache
  • superblock: file system - specific information regarding the FS layout

VFS on Disk (other than the dentry)

  • file == data blocks on disk
  • inode == track files’ blocks, also resides on disk in some blocks
  • superblock == overall map of disk blocks
    1. inode blocks
    2. data blocks
    3. free blocks

Example - ext2: Second Extended Filesystem

inodes

  • Limits file size

inodes with indirect pointers

  • Benefits: small inode could map to large file size
  • Downside: slow down file access
    • e.g. direct ptr => 2 disk access, while double indirect ptr => up to 4 disk accesses

Disk Access Optimizations

Caching/buffering

  • Reduce # disk accesses
  • In main memory
  • Read/write from cache
  • Periodically flush to disk

I/O Scheduling

  • Reduce disk head movement
  • Maximize sequential v.s. random access
  • e.g., write block 25, write block 17; if head is at 15 => write 17, 25; they are still in memory (i.e. will be lost if crashes)

Prefetching

  • Increases cache hits
  • Leverages locality
  • e.g., read block 17 => also read 18, 19

Journaling/logging

  • Reduce random access
  • “describe” write in log: block, offset, value…
  • Periodically apply updates to proper disk locations
